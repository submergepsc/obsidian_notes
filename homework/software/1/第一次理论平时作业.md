## 一、简答题
### 1. 需求工程包括哪些活动？需求工程为什么重要？
需求工程主要包括以下几个活动：
1. **需求获取（Elicitation）**：从用户、利益相关者那里收集需求
2. **需求分析（Analysis）**：整理、分类、发现冲突和不一致
3. **需求规格说明（Specification）**：用文档形式准确描述需求
4. **需求验证（Validation）**：确认需求是否正确、完整、可行
5. **需求管理（Management）**：对需求的变更进行控制和跟踪
**重要性：**
需求工程是软件开发的基础。如果需求出错，后面设计、编码再好也没用。
有研究表明，大部分软件失败都源于需求阶段的问题，例如需求不清晰、频繁变更等。
 
### 2. 为什么用户参与不足？如何解决？
**用户参与不足的原因：**
1. 用户时间有限，不愿投入精力
2. 用户不懂技术，不知道如何表达需求
3. 沟通成本高（语言、认知差异）
4. 用户认为这是开发人员的事情
5. 项目组织不重视用户参与
**解决方法：** 
1. 提供清晰的沟通方式（原型、图示等）
2. 安排固定沟通时间（例如定期评审）
3. 使用简单易懂的语言，避免技术术语
4. 提高用户参与的积极性（例如让用户看到成果）
5. 设立“用户代表”参与项目
### 3. 什么是场景？什么是用例？如何描述用例？
**场景（Scenario）：**
描述用户在特定情境下与系统交互的具体过程，是一个故事。
**用例（Use Case）：**
描述系统如何响应用户行为的一组完整功能，用于表达系统功能需求。
**用例的描述通常包括：** 
1. 用例名称
2. 参与者（Actor）
3. 前置条件
4. 基本流程（主流程）
5. 扩展流程（异常情况）
6. 后置条件
  核心区别： 
* 场景 = 一个具体故事
* 用例 = 一类行为的抽象描述
### 4. 什么是动态交互建模？有哪些UML图？
**动态交互建模：**
描述系统在运行过程中，各对象之间如何交互和变化。
常见UML动态图包括： 
#### 1）顺序图（Sequence Diagram） 
* 描述对象之间**按时间顺序**的交互
* **优点：** 清晰展示调用顺序
* **缺点：** 不适合复杂分支逻辑 
#### 2）通信图（Communication Diagram） 
* 强调对象之间的关系和消息传递
* **优点：** 结构清晰
* **缺点：** 时间顺序不明显 
#### 3）状态图（State Diagram） 
* 描述对象的状态变化
* **优点：** 适合复杂状态系统
* **缺点：** 不适合描述整体流程 
#### 4）活动图（Activity Diagram） 
* 类似流程图，描述业务流程
* **优点：** 易理解

* **缺点：** 细节交互不够清晰
## 二、分析题
### 1. 在AI时代，试分析计算机系统改变世界的能力和潜力，给出合理性的解释。
AI时代计算机系统改变世界的能力，计算机系统的能力已经从“工具”转变为“智能决策参与者”。
**主要体现在：**
1. **自动化能力增强**
   * 替代重复性工作（如客服、数据处理）
2. **数据分析能力提升**
   * AI可以从海量数据中发现规律
3. **智能决策支持**
   * 如医疗诊断、金融预测
4. **创造能力（生成式AI）**
   * 文本、图像、代码生成
5. **人机协作模式改变**
   * 从“人操作机器”变为“人+AI协作”
 合理性：
* 计算能力提升
* 数据爆炸
* 算法进步（如深度学习）
### 2. 三种需求获取方法对比 

| 方法   | 优点   | 缺点   | 适用场景   |
| ---- | ---- | ---- | ------ |
| 调查问卷 | 覆盖面广 | 深度不足 | 用户数量多  |
| 头脑风暴 | 创意多  | 容易发散 | 早期需求探索 |
| 面谈   | 深度高  | 成本高  | 关键用户分析 |

 总结：
* **问卷**：量大
* **面谈**：深度
* **头脑风暴**：创新
### 3. 沟通的重要性及团队建设
**为什么沟通重要：**
1. 需求本质是“理解用户”
2. 减少误解和需求偏差
3. 提高开发效率
4. 降低项目风险
**组建团队时注意：**
1. 成员角色清晰（分析师、开发、用户代表）
2. 技术与业务能力结合
3. 沟通能力强
4. 团队规模适中
**如何维持有效沟通：**
1. 定期会议（站会、评审）
2. 使用统一工具（文档、原型）
3. 明确沟通规范
4. 保持信息透明
5. 建立反馈机制
